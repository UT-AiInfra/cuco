<DESCRIPTION>
The current program has a critical bottleneck: the compute stream waits for dispatch_wait_done before starting ANY compute, even though self-data is already available after self-copy completes.

Looking at the timing:
- Rank 0: dispatch_wait takes 0.7ms, then compute takes ~134ms
- Rank 1: dispatch_wait takes 1.9ms, then compute takes ~45ms, then combine_wait takes 95.7ms

The key insight is that self-data (from self-copy) doesn't need to wait for remote data to arrive. We can:
1. Start self-compute immediately after self-copy completes (on compute_stream)
2. Only wait for dispatch_wait_done before processing remote data

This allows self-compute (~67ms for Rank 0's 3072 self-tokens) to overlap with the network transfer, potentially hiding the dispatch latency entirely.

The change restructures the pipeline to:
1. Issue dispatch PUT on comm_stream
2. Do self-copy on compute_stream
3. Immediately start self-compute (dequant→gemm1→swiglu→gemm2) for self-data on compute_stream
4. Meanwhile, comm_stream waits for remote data
5. After self-compute, compute_stream waits for dispatch_wait_done, then processes remote data
6. This allows self-compute to overlap with network transfer
</DESCRIPTION>
