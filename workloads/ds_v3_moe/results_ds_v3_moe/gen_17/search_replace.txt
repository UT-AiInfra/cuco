overlap_dispatch_wait_with_self_compute

<DESCRIPTION>
Looking at the per-phase timings, the current program launches dispatch_wait immediately after dispatch_put on the comm_stream, but self_compute on compute_stream doesn't start until after self_copy_dispatch completes. The key insight is that self_compute processes LOCAL data (from self-copy), not remote data. Therefore, self_compute can run CONCURRENTLY with dispatch_wait.

The best-performing prior program (83.88 score) had dispatch_wait taking 67.1ms which matched self_compute at 67.0ms - this suggests they were running in parallel and the wait was just polling while compute ran.

The current program shows:
- Rank 0: dispatch_wait=0.69ms, self_copy=0.76ms, self_compute=67.2ms (sequential)
- The dispatch_wait finishes quickly (0.69ms) because the PUT kernel returns fast

The issue is that self_copy_dispatch (0.76ms) is blocking the compute stream unnecessarily. We should:
1. Move self_copy to run concurrently with dispatch_put (not after dispatch_wait)
2. Start self_compute immediately after self_copy, without waiting for dispatch_wait
3. Only wait for dispatch_wait before remote_compute

This should hide the dispatch communication latency behind self_compute.
</DESCRIPTION>
