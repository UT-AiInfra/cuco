<DESCRIPTION>
Looking at the per-phase timings, I notice that `dispatch_wait` takes almost exactly the same time as `self_compute` (22.5ms vs 22.5ms for rank 1, 67.1ms vs 67.1ms for rank 0). This suggests that the dispatch_wait is NOT actually overlapping with self_compute - they're running sequentially.

The issue is in the current code structure: the `ginDispatchWaitKernel` is launched on `comm_stream` AFTER the self_compute kernels are launched on `compute_stream`, but the timing shows they're not truly concurrent.

The key insight from the best-performing program (score 83.88) is that it achieves the same ~118ms time. Looking at the phase timings more carefully:
- dispatch_wait time ≈ self_compute time suggests the network transfer completes around when self_compute finishes
- This is actually good - it means the network transfer IS happening during self_compute

The real bottleneck is the `combine_wait` phase (2.4ms + 7.2ms = ~9.6ms total across ranks). This is pure waiting time that could potentially be hidden.

However, the safest incremental change is to ensure the dispatch_wait kernel is launched IMMEDIATELY after dispatch_put (before self_copy and self_compute), so the wait polling starts as early as possible. This ensures maximum overlap between the network transfer completion detection and the self_compute work.

Currently the code launches:
1. dispatch_put on comm_stream
2. self_copy on compute_stream
3. self_compute on compute_stream
4. dispatch_wait on comm_stream (AFTER self_compute is queued)

The fix: Launch dispatch_wait immediately after dispatch_put, before any compute_stream work. This ensures the wait kernel starts polling immediately while self_compute runs.
